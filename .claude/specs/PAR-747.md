# Spec for Multi-Tenant Checkout

branch: feature/PAR-747-Multi-Tenant-Checkout

## Summary

The Merchant Portal (an internal system for merchant account management) needs to embed the demo-shop checkout frontend. When doing so, it will pass `merchant_ref` and `assets_key` as GET parameters to the frontend page. These values identify which merchant's checkout experience should be rendered and which SeQura assets to load.

A global API account (with permissions across all merchant references) is already configured in `.env`. The backend must detect the presence of `merchant_ref` and `assets_key`, persist them for the entire checkout lifecycle, and use them when calling the SeQura API — replacing the default single-merchant credentials and configuration where applicable.

IPN (webhook) callbacks must carry the `merchant_ref` back to the backend via `notification_params` so that IPN handling can resolve the correct merchant context without relying on session state.

When `merchant_ref` and `assets_key` are absent, the system must behave exactly as it does today (backward-compatible fallback to the default checkout flow).

## Functional Requirements

- The checkout page must accept `merchant_ref` and `assets_key` as URL GET parameters.
- When both parameters are present, the backend must persist them (tied to the current checkout session) so they are available throughout solicitation, order creation, and IPN handling.
- When either parameter is absent, the backend must fall back to the existing default checkout flow with no change in behavior.
- During solicitation and order creation, if a `merchant_ref` is active, the global API account credentials must be used and the `merchant_ref` must be forwarded to the SeQura API.
- The `assets_key` must be forwarded to the frontend so SeQura's widget/asset CDN loads the correct assets for the merchant.
- The `merchant_ref` must be embedded in the `notification_params` of the SeQura order so that it is returned to the backend when SeQura fires the IPN webhook.
- The IPN handler must extract `merchant_ref` from `notification_params` and use it (with the global API credentials) to process the webhook correctly.
- The global API credentials must only be used when a `merchant_ref` is explicitly present; default credentials must be used otherwise.
- No existing checkout behavior (for users not coming from the Merchant Portal) must be altered.

## Possible Edge Cases

- `merchant_ref` is provided but `assets_key` is missing (or vice versa) — the system must treat this as an incomplete multi-tenant request and fall back to the default flow, or reject the request explicitly.
- A tampered or invalid `merchant_ref` is supplied — the SeQura API will reject it; the error must be handled gracefully without leaking internal details.
- Session expiry between steps — if the persisted `merchant_ref`/`assets_key` is lost mid-flow, subsequent API calls may fail; the error response should be meaningful.
- IPN arrives without `merchant_ref` in `notification_params` (e.g., for orders created before this feature) — must fall back to default IPN handling.
- Concurrent sessions with different `merchant_ref` values — each session must be isolated; one session must not leak context to another.
- The global API account credentials are missing or misconfigured in `.env` — the system must fail fast with a clear error rather than silently using the wrong credentials.

## Acceptance Criteria

- A checkout flow initiated with valid `merchant_ref` and `assets_key` GET parameters completes end-to-end (solicitation → payment selection → order confirmation → IPN) using the global API account and the correct merchant context.
- A checkout flow initiated without those parameters behaves identically to the current production flow.
- The IPN handler correctly resolves the merchant from `notification_params` and processes the webhook without error.
- No cross-session contamination: two simultaneous sessions with different `merchant_ref` values each complete independently and correctly.
- If `merchant_ref`/`assets_key` are only partially provided, the system falls back to the default flow (or returns a clear error), and does not partially apply multi-tenant logic.

## Open Questions

- Should partial parameter presence (only `merchant_ref` or only `assets_key`) be a silent fallback to default, or an explicit error returned to the caller?
- How should the persisted `merchant_ref`/`assets_key` be stored — in the PHP session, in the encrypted JSON repository, or as part of the order record?
- Is there a whitelist of valid `merchant_ref` values that the backend should validate against, or is validation deferred entirely to the SeQura API?
- Should the Merchant Portal embed the checkout in an iframe, or navigate to it directly? This affects CSRF and Origin/Referer header handling in `SecurityMiddleware`.
- Are there any additional per-merchant configuration values (beyond `merchant_ref` and `assets_key`) that the Merchant Portal may need to pass in the future?

## Testing Guidelines

Follow the CLAUDE.md testing guidelines and create meaningful tests for the following cases, without going too heavy:

- Default flow (no GET params) passes through unchanged — solicitation and IPN use default credentials and config.
- Multi-tenant flow with valid `merchant_ref` + `assets_key` — correct credentials and merchant context are used at each step.
- IPN with `merchant_ref` in `notification_params` routes to the correct merchant handler.
- IPN without `merchant_ref` in `notification_params` falls back to default handling.
- Partial parameter presence triggers fallback (or explicit error) without applying multi-tenant logic.

## Clarifications

1. **Partial parameter handling:** Silent fallback to default flow. If only one of `merchant_ref`/`assets_key` is present, treat as if neither was provided.
2. **Storage mechanism:** Use `$_SESSION` to persist `merchant_ref` + `assets_key` for the checkout lifecycle. Session is already initialized in `index.php` with secure cookie settings.
3. **Validation of `merchant_ref`:** Deferred entirely to the SeQura API — no local whitelist.
4. **Iframe vs. direct navigation:** Direct navigation. No iframe embedding, so no changes needed to `SameSite` cookie policy, CSRF, or Origin validation.
5. **Future extensibility:** No additional per-merchant config values expected now, but the design should allow easy addition of new values (use a structured "tenant context" in session rather than individual keys).
6. **Merchant-to-cart mapping for IPN:** Option A — store `merchant_ref` alongside order data in `DemoSeQuraOrderRepository` (keyed by `cartId`, using `_merchant_ref` prefix convention). `DemoMerchantDataProvider` reads it back via the repository to populate `notification_params`. This decouples IPN handling from session state.

## Analysis

### Affected Files

**Backend — Controllers:**
- `src/backend/src/Controllers/PageController.php` — extract `merchant_ref` + `assets_key` from GET params, store in session, pass `assets_key` to view (override credentials-based value when present)
- `src/backend/src/Controllers/CheckoutController.php` — read tenant context from session; when `merchant_ref` present, skip country-code credential lookup and use session values; store cart→merchant mapping in order repo before solicitation call
- `src/backend/src/Controllers/WebhookController.php` — extract `merchant_ref` from IPN payload notification params; use it for webhook handler context

**Backend — Platform Adapters:**
- `src/backend/src/Platform/DemoMerchantDataProvider.php` — inject `SeQuraOrderRepositoryInterface`; implement `getNotificationParametersForCartId()` to return `['merchant_ref' => $ref]` when a mapping exists; same for `getEventsWebhookParametersForCartId()`

**Backend — Repositories:**
- `src/backend/src/Repository/DemoSeQuraOrderRepository.php` — add `setMerchantContext(string $cartId, string $merchantRef)` and `getMerchantRef(string $cartId): ?string` methods

**Backend — Bootstrap & Config:**
- `src/backend/src/Bootstrap.php` — update `DemoMerchantDataProvider` registration to inject the order repository
- `src/backend/views/checkout.php` — pass `merchant_ref` as a data attribute on `<sequra-checkout>` so frontend can forward it in API calls if needed

**Frontend:**
- `src/frontend/src/components/SeQuraCheckout.js` — read optional `merchant-ref` attribute; include `merchant_ref` in solicitation POST payload so the backend can associate it with the cart (belt-and-suspenders alongside session)

### Risks & Concerns

1. **Session loss mid-flow:** If the PHP session expires between page load and solicitation, the tenant context is lost. Mitigation: the 4-hour session TTL (matching CSRF TTL) is sufficient for a checkout flow. If session is lost, the solicitation falls back to default flow — acceptable degradation.
2. **Static repository cache:** `DemoBaseRepository` uses a static class-keyed cache. Two concurrent requests with different `merchant_ref` values in the same PHP process could theoretically share cached data. Mitigation: PHP-FPM processes are request-isolated (each request gets its own process memory), so static caches are not shared across requests.
3. **IPN timing:** The cart→merchant mapping must be stored *before* the solicitation API call, because the integration-core calls `getNotificationParametersForCartId()` during order creation. If stored after, the params would be empty.
4. **Backward compatibility:** All changes are additive. When `merchant_ref`/`assets_key` are absent, every code path falls through to existing behavior. No existing behavior is modified.

### Decisions

1. **Tenant context structure in session:** Use `$_SESSION['tenant']` as an associative array (`['merchant_ref' => '...', 'assets_key' => '...']`) to allow easy future extension.
2. **Cart-to-merchant mapping:** Stored in `DemoSeQuraOrderRepository` using `_merchant_ref` key (following existing `_status`, `_sq_state` convention for app-specific fields).
3. **Frontend attribute:** Add `merchant-ref` attribute to `<sequra-checkout>` component. Frontend reads it and includes in solicitation payload. This ensures the backend can always associate the cart with the merchant, even if the session value was set on a different route.
4. **No SecurityMiddleware changes:** Direct navigation means no iframe/CORS/SameSite changes needed.
